什么是 Pull Request

你改的代码
怎么让别人审核？

PR = 发起一个请求，请项目维护者拉取（Pull）你的代码到主分支

类比：编辑部投稿
记者写稿 → 编辑审稿 → 通过后发表。你的 PR 就是"投稿"，维护者就是"编辑"。
讲师提示：很多学员第一次用 GitHub 就是"直接改代码推到 main 分支"。要强调：PR 是团队协作的必备流程，不是可选项。

PR 工作流
Step 1
Fork 仓库
把项目复制到自己账号
Step 2
开新分支
git checkout -b feature/xxx
Step 3
提交代码
git commit + git push
Step 4
发起 PR
GitHub 网页上点 New Pull Request
讲师提示：演示一遍完整的 PR 流程。重点：为什么不能直接推 main 分支——因为维护者的 main 可能比你新，直接推会覆盖。

COLLABORATION
Code Review 是什么

你的代码
要被谁看过？

没有 Review
直接合并
bug 进入主分支，影响所有人
  • 代码风格不统一
  • bug 无人发现就上线
  • 知识只在一个人脑子里
有 Review
团队一起看
PR 需要至少 1 人 Approve 才能合并
  • 早发现 bug，减少返工
  • 知识在团队内传播
  • 代码质量团队共同负责
讲师提示：强调 Code Review 的核心价值不是"挑错"，而是"知识传递"。很多团队把 Review 当审批流程，其实它是最有效的学习机制。

COLLABORATION
好的 PR 描述

让 Reviewer 5分钟内看懂

必须写
  • 1. 这个 PR 解决什么问题？
  • 2. 怎么测试的？（截图/步骤）
  • 3. 有没有影响其他功能？
不要写
  • ✗ "修复了一些bug"
  • ✗ "更新了代码"
  • ✗ 不写，等 reviewer 自己看
模板：What / Why / How — 这个 PR 做了什么、为什么做、怎么测试的
讲师提示：给学员展示一个真实的好 PR 描述例子。强调：好的描述节省的是所有人的时间——包括你自己（被 merge 更快）。

COLLABORATION
协作礼仪

Review 别人代码的正确姿势

建议语气
"这里可以考虑..."
而不是"这里错了"
具体指出
指出行号+原因
附上参考链接或文档
分类标签
nit / must / suggestion
nit = 小问题，must = 必须改，suggestion = 建议
讲师提示：这是软技能。很多技术人 Review 时只会说"不行"。教他们用"三明治"反馈法：肯定+建议+鼓励。

COLLABORATION
新手常犯的错误
直接在 main 分支改代码
应该：开新分支 → PR → Review → 合并
PR 描述写"update"
应该：写清楚 What / Why / How
一个 PR 塞 20 个改动
应该：小步提交，每个 PR 只做一件事
不写测试就说"能跑"
应该：截图/录屏证明功能正常
讲师提示：让学员分享自己见过的"反面案例"。教学效果最好的是他们自己踩过的坑。
